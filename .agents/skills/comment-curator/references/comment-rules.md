# Comment Rules

- Comments are for future human readers.
- Good comments explain why, constraints, examples, context, or non-obvious tradeoffs.
- Bad comments repeat the code in weaker prose.
- A better variable, method, or class name often removes the need for a comment.
- Stale comments are worse than absent comments.
- TODO, HACK, and FIXME notes should not become excuses for permanent weak code.
- Multi-line comments should be visually easy to scan.
- Configuration comments should include concrete examples when the expected value shape is unclear.
- Commented-out code is clutter. Remove it unless the project has an explicit temporary migration convention.
